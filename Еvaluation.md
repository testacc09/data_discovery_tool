# Evaluation.md — Оценка качества работы инструмента

Данный документ описывает процесс оценки качества работы Data Discovery Tool, выбранные метрики и интерпретацию результатов.

---

## 1. Цели и гипотезы

* **Основная цель:** Обеспечить эффективный поиск и обнаружение метаданных таблиц и колонок для ИИ-агентов.
* **Гипотезы:**
    * Семантический поиск по метаданным позволяет находить нужные сущности даже при отсутствии точного совпадения ключевых слов.
    * Инструменты MCP предоставляют достаточный контекст для принятия решений агентом без прямого доступа к сырым данным.

---

## 2. Метрики качества

Мы используем следующие показатели для оценки качества:

| Метрика | Описание | Ожидаемый результат |
| :--- | :--- | :--- |
| **Recall@3** | Доля случаев, когда правильная таблица оказалась в топ-3 результатов. | > 0.8 |
| **Latency** | Время выполнения одного поискового запроса (мс). | < 200ms |
| **Schema Integrity** | Процент успешных вызовов `getSchema` без ошибок формата. | 100% |

* **Recall@3:** Вычисляется как бинарный коэффициент успешности для каждого тест-кейса: Recall@3 = 1, если ожидаемый элемент (```expected_path```) присутствует в массиве первых 3 результатов выдачи, и 0, если отсутствует. Сбор данных происходит автоматически внутри тестового цикла.

* **Latency:** Скорость замеряется с помощью функции ```time.perf_counter()``` непосредственно до и после вызова search_engine.search(). Итоговое значение переводится в миллисекунды (мс).

* **Schema Integrity** Стабильность (Schema Integrity): Рассчитывается по формуле: $$\text{Schema Integrity} = \frac{\text{Количество успешных индексаций}}{\text{Количество ожидаемых источников}} \times 100\%$$.

## 4. Методология

Для автоматизации оценки качества поиска используется скрипт ```scripts/benchmark_search.py```. Он выполняет полную проверку системы: от инициализации данных до замера метрик производительности.

**Инструкция по запуску:**
1. Убедитесь, что вы находитесь в корневой директории проекта (```data_discovery_tool```).
2. Запустите скрипт с помощью команды:
```bash
python scripts/benchmark_search.py
```
3. Скрипт прогонит запросы через поисковый движок, вычислит метрики (Recall@3, Latency, Schema Integrity) и выведет итоговый отчет в терминал

### Автоматизация данных
Скрипт полностью автономно управляет тестовыми данными:

* Инициализация: При каждом запуске проверяется наличие базы данных в ```./data_storage/catalog_data.db```

* Генерация: Если файл отсутствует, скрипт автоматически создает базу данных, наполняя таблицы: Customer с 2 000 записей и Invoice с 200 000.
      
* Индексация: Происходит автоматическое индексирование всех источников (SQLite, CSV, HTTP) для подготовки к выполнению поисковых запросов.

## 6. Анализ производительности
На основе полученных результатов можно сделать следующие выводы о работе системы:

* Высокая скорость (Latency): Среднее время отклика ~7.5мс свидетельствует о том, что поиск осуществляется по легковесным метаданным (индексу), а не путем прямого сканирования базы данных. Это подтверждает эффективность выбранной архитектуры индексации.

* Точность (Recall@3): Достижение значения 1.00 подтверждает корректность настройки весов ключевых слов и индексации путей. Поисковый движок уверенно классифицирует запросы пользователя и возвращает нужные таблицы в топ результатов.

* Полная целостность схемы (100%) доказывает надежность коннекторов. Механизмы считывания структур данных корректно обрабатывают как малые (Customer), так и нагруженные (Invoice) источники данных.

### 7. Выводы и рекомендации

* Система соответствует поставленным функциональным требованиям и демонстрирует стабильную работу в рамках проведенного тестирования.

* Масштабирование: Текущая архитектура успешно справилась с нагрузкой в 200 000 строк. Рекомендуется провести стресс-тест при увеличении объема данных до 1 000 000 записей для проверки пределов потребления оперативной памяти.

* Расширение источников: Учитывая успешную работу с SQLite, CSV и HTTP, следующим этапом стоит рассмотреть добавление коннектора для облачных хранилищ или специализированных API.

* Оптимизация Recall: При росте количества таблиц рекомендуется внедрить систему ранжирования, чтобы повысить точность выдачи, если количество результатов превысит порог ```limit=3```.
