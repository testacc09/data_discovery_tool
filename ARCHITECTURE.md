### Основные модули:

* **Интерфейс пользователя (`ui/app.py`):** Легковесное приложение на базе Streamlit. Оно запускает фоновое сканирование схем данных, обрабатывает поисковые запросы пользователя и отображает интерактивный предпросмотр найденных таблиц.
* **Поисковый движок (`search/engine.py`):** Центральный интеллектуальный модуль. Использует библиотеку `fastembed` для превращения поисковых запросов в векторы, рассчитывает косинусное сходство через матричные операции `numpy` и комбинирует полученный результат с поиском по точному совпадению подстрок.
* **Пул коннекторов (`connectors/`):** Набор изолированных адаптеров для различных источников данных. Каждый коннектор строго следует общему интерфейсу, что позволяет извлекать метаданные схем без полной загрузки тяжелых файлов в оперативную память.
* **Векторное хранилище (`data_storage/embeddings_cache.json`):** Локальный JSON-файл, выполняющий роль кэша. Он сохраняет рассчитанные эмбеддинги метаданных, чтобы полностью исключить повторные вычисления нейросети при перезапуске приложения.

### Технологический стек и инструменты:

* **Язык:** Python 3.10+
* **Интерфейс:** Streamlit
* **Поисковый движок:** FastEmbed
* **Эмбеддинговая модель:** `BAAI/bge-small-en-v1.5` 
    * *Выбрана как высокопроизводительная модель с малым весом, обеспечивающая отличную точность семантического поиска для коротких метаданных.*
* **Математический аппарат:** NumPy (расчет косинусного сходства)

## 2. Ключевые интерфейсы и контракты

Для обеспечения масштабируемости каждый коннектор наследуется от базового класса и реализует строгий поведенческий контракт. Благодаря этому `SearchEngine` может работать с любым новым типом источника без изменения своего внутреннего кода.

```python
from typing import List, Dict, Any, Optional

class BaseConnector:
    def __init__(self, source_id: str, connection_string: str):
        self.source_id = source_id
        self.connection_string = connection_string

    def get_source_id(self) -> str:
        ...

    def discover_schemas(self) -> List[Dict[str, Any]]:
        ...

    def get_table_preview(self, path: str, limit: int = 100) -> Optional[Any]:
        ...
```

### Спецификация формата схемы:

Метод `discover_schemas()` каждого коннектора обязан возвращать унифицированную структуру данных. Это гарантирует, что поисковый движок сможет проиндексировать метаданные независимо от того, пришли они из реляционной БД, локальной папки или внешнего API.

```json
[
  {
    "table_name": "orders",
    "row_count": 5001,
    "columns": [
      {"name": "order_id", "type": "INTEGER"},
      {"name": "customer_name", "type": "TEXT"},
      {"name": "total", "type": "REAL"}
    ]
  }
]
```

При запуске веб-интерфейс запрашивает настроенный движок. Если локальный кэш эмбеддингов уже сформирован, тяжелые нейросетевые вычисления полностью пропускаются.

## 4. Архитектурные компромиссы

* **Матрица в памяти вместо выделенной Vector DB:**

Решение: Выбран кастомный поиск на NumPy с хранением векторов в JSON-файле вместо развертывания ChromaDB/Milvus.

Обоснование: Для задач Data Discovery по метаданным объем векторов исчисляется тысячами, что легко помещается в ОЗУ. Это избавляет проект от лишних инфраструктурных контейнеров.

* **Индексация метаданных вместо индексации содержимого строк:**

Решение: Система семантически анализирует исключительно структуру таблиц, типы данных и названия полей, намеренно игнорируя построчный разбор ячеек.

Обоснование: Этот подход снижает объем дисковой памяти под кэш и предотвращает блокировки со стороны баз данных при старте. Инструмент остается средством обнаружения (```discovery```).

* **Гибридный режим многопоточности (Разделение индексации и поиска):**

Решение: На этапе фоновой индексации больших пакетов данных fastembed задействует параллельные потоки процессора через асинхронные вызовы ```asyncio.to_thread```. Однако при вычислении эмбеддинга одиночного поискового запроса внутри метода ```.search()``` используется строго однопоточный режим (```threads=1```).

Обоснование: При первоначальном сканировании источников многопоточность значительно ускоряет обработку сотен названий колонок. Напротив, при обработке одного введенного пользователем слова создание и синхронизация пула потоков в ОС занимают больше времени, чем сама математика вектора. Ограничение до ```threads=1``` на этапе поиска убирает системные задержки и делает UI мгновенным.
